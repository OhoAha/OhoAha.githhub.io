Security in Agile


INTRODUCTION
Agile development becomes more popular due to its fast development, flexible, high customer satisfaction, and simultaneously can build a quality product. Regrettably, agile SDLC's fast nature and limited time make some agile organizations or developers neglecting the importance of integrating security throughout project development. There are a lot of issues and challenges that become the reason for many agile organizations leaving security behind despite knowing the significance of applying security in their product.Therefore, there are several requirements for security to properly integrate into the agile process.


Requirements for Security to properly integrate into the Agile process
1. Security approach must be adaptive to the agile software development methods and not hinder the development process.
Requirements for Security to properly integrate into the Agile process
2. Security approach that need to be integrated successfully with agile development methods should offer concrete guidance and tools at all phases of development such as from requirements phase to deployement phase.
Requirements for Security to properly integrate into the Agile process
3. A successful security element should be able to adapt rapidly to ever-changing requirements owing to a fast-paced business environment, including support for handling several incremental iterations.

AGILE SECURITY MANIFESTOES

1. Rely on developers and testers more than security specialists.

2. Secure while we work more than after we’re done.

3. Implement features securely more than adding security features.

4. Mitigate risks more than fix bugs.

AGILE SECURITY PRACTICES

Agile security practices for all team:

  1. Review and document the significant threat and huge risk as needed while discarding intricate threat modeling processes and related software.Then, if the process fit the agile model, the team can swiftly implement the incremental change to the threat model, so that, they can proceed to development tasks.

  2. Install a static analysis tool on each developer’s device. This tool can give immediate feedback for the developer to verify a code flaw and malicious code in the program.

  3. Configure dynamic analysis tools to frequently test the software running in the development staging and quality assurance environment. Therefore, for each short sprint, the result of the automated test will be reduced to the code committed only, and it must regularly monitor by the project team. It made the security agile during the development test easy to manage.

  4. Add any identified security issues that prioritize a list of bugs enhancements and requirements during the security tests to the product backlog. Usually, item in the backlog is written as user stories.

Sometimes, there are security issues that can be missed by the automated test during each sprint. So, it is best to have an expert security source code review and penetration test to identify the problem. Then, the problem will be added to the backlog. After that, when the security sprint is completed, do another penetration test to validate the fix and correction before approving the production release.



The security SME should :

  1. Provide continuous security for each sprint through integration and monitoring to adapt it to the agile model.

  2. Adjust the task like application design attack surface analysis, threat modeling, and security architecture review to be as lightweight as possible. For example, at the beginning of the iteration, the team can do a short technical meeting or a simple questionnaire about the security task.

  3. Help the development team write unit tests for security-specific cases. Thus, it will make the development team can execute and monitor the unit test automatically.

  4. Review and identify the vulnerabilities and risky security elements like password management, validation libraries, encryption, and authentication. The experienced developer should modify these elements and monitor it on the build server for the code change.

  5. Assist the project team with examining the abuse stories in the backlog or the security-specific functional abuser and the writing testing.

  6. Examine the current state of the application security when at the end of the sprint meeting.

At the end of the sprint, conduct a final security review. It is depending on the risk tolerance and the severity of the application. The product is ready to be released after enough secure iterations and feedback loops with the customer. Then, the product can be assumed to be secure and safe.
